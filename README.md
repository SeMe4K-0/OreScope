# OreScope — автоматическая классификация руд по панорамным OM-изображениям шлифов

> ### Живое демо: **https://orescore.ru.tuna.am/**
> Дашборд развёрнут и работает прямо сейчас — засеян полным датасетом `ore_data/` и
> обрабатывает его фоновой очередью. Поток: **Общий анализ → Образец шлифа →
> Экспертная проверка → Загрузка**. Ссылка временная (туннель на локальную машину),
> при недоступности см. раздел «Docker» ниже для локального запуска.

Решение хакатона Норникель: end-to-end система, которая по микрофотографиям полированных
шлифов (от единичных кадров до панорам 300+ Мп) сегментирует сульфиды, разделяет их на
**обычные** и **тонкие** срастания, детектирует **тальк** и классифицирует руду по
экспертному правилу:

- зоны оталькования > 10 % площади (и тальковая фаза > 5 %) → **оталькованная**;
- иначе, преобладают обычные срастания → **рядовая**;
- иначе → **труднообогатимая**.

Результат интерпретируем: цветовая маска (зелёный = обычные, красный = тонкие,
синий = тальк), карта уверенности, таблица метрик, текстовое заключение,
экспорт CSV / PDF / GeoJSON и JSON-лог параметров для воспроизводимости.

## Два числа для талька — почему

Тальк в шлифах — **рассеянная тёмная фаза**: сотни мелких частиц, а не сплошные поля.
Эксперт же обводил на обучающих фото **зоны** оталькования (внутри них есть и матрица,
и сульфиды), и порог «10 %» исторически относится к зонам. Поэтому система репортит оба
числа: **тальк (фаза)** — честная площадь самой фазы, и **зоны оталькования** —
реконструкция зон из фазы (компоненты локальной плотности с заполнением ≥ 12 %;
калибровка по 39 экспертным парам, систематика −1.9 п.п.). Правило 10 % применяется
к зонам; фаза > 5 % — анти-шумовое условие (свип `scripts/sweep_rule.py`).

## Архитектура

```
Изображение (TIFF/PNG/JPEG, до 300+ Мп)
  └─ Предобработка: нормализация цветового профиля к домену обучения (Reinhard, LAB,
     полосами для панорам) → маска валидных пикселей → медианный фильтр (царапины) →
     выравнивание освещения → серый баланс → CLAHE
  └─ Сульфиды: адаптивный порог яркости (Otsu + робастная страховка)
  └─ Срастания: CNN (resnet18, weak labels, ~1050 фото) поверх зёренной морфологии
  └─ Тальк: U-Net (resnet34, ImageNet) — маски «рассеянная фаза» (39 уточнённых
     экспертных пар ч1 + 79 проверенных псевдомасок ч2 + 120 негативов, ignore-зоны
     вне лосса); тайлы 1024px, косинусное смешивание, TTA
  └─ Зоны оталькования из фазы (плотность + заполнение) → экспертное правило
     → флаги пограничных случаев («требуется проверка»)
```

Единая точка входа — `src/pipeline.py:analyze_image`; её используют дашборд,
CLI и скрипты оценки.

## Быстрый старт (локально)

Веса рабочих моделей (`models/talc_best.pt`, `models/grain_cnn.pt`) хранятся через
**Git LFS** прямо в репозитории — воспроизведение не требует переобучения.

```bash
git lfs install && git lfs pull          # подтянуть веса (если клонировали без LFS)
pip install -r requirements.txt          # Python 3.11, GPU опционален
python app.py                            # дашборд на http://localhost:7860 (ORESCOPE_PORT)
```

Дашборд — SPA на базе дизайна AlloyScope (`frontend/AlloyScope.dc.html` + `support.js`),
API «партии → образцы»: при старте весь `ore_data/` засевается партиями по эталонным
классам (+ панорамы) и обрабатывается фоновой очередью. Поток работы:
**Загрузка → Экспертная проверка (отдельный экран: вердикт/класс по каждому
обработанному образцу) → Образец шлифа**; «Общий анализ» — статистика по партии.

CLI-анализ (одно фото или папка, пакетный режим):

```bash
python scripts/analyze.py --input "ore_data/Панорамы" --out outputs/panoramas --um-per-px 1.75
```

## Docker

Данные и выходы в образ не попадают (`.dockerignore`); веса моделей (`models/*.pt`)
запекаются в образ, если лежат в репозитории на момент сборки, иначе монтируются томом.

```bash
# сборка (CPU-образ, ~2.5 ГБ; веса в репо -> внутри образа)
docker build -t orescope .

# запуск: дашборд на http://localhost:7860, результаты наружу через том
docker run -d --name orescope -p 7860:7860 -v ./outputs:/app/outputs orescope

# если веса НЕ запечены в образ — смонтировать их томом
docker run -d -p 7860:7860 -v ./models:/app/models -v ./outputs:/app/outputs orescope

# пакетный CLI-прогон внутри контейнера (данные подключаются томом read-only)
docker run --rm -v ./ore_data:/app/ore_data:ro -v ./outputs:/app/outputs orescope \
  python scripts/analyze.py --input "ore_data/Панорамы" --out outputs/panoramas
```

GPU-вариант: заменить базовый образ на `pytorch/pytorch:2.3.0-cuda12.1-cudnn8-runtime`
(строку установки CPU-torch убрать) и запускать с `--gpus all`. Локальное развёртывание —
данные не покидают машину; healthcheck на `/api/batches` встроен.

## Обучение и калибровка (воспроизводимость)

| Скрипт | Назначение |
|---|---|
| `scripts/compute_ref_profile.py` | референсный LAB-профиль домена ч1 для цветонормализации (`config.yaml → preprocessing.color_norm`) |
| `scripts/make_talc_masks.py` | синие контуры эксперта → маски тальковой **фазы** (замыкание разрывов, `bg_points`/`exclude` в `data/talc_masks_overrides.json`, уточнение «зона → рассеянная фаза» `--k-dark 0.6`, ignore-маски в `data/talc_ignore/`, contact-sheets в `outputs/talc_review/`) |
| `scripts/train_talc.py` | U-Net талька; Dice+CE c `ignore_index=255`; сплит по изображениям; `--negatives N --neg-domains ch1\|all` (негативы), `--extra-dir data/talc_masks_ch2` (псевдомаски ч2), `--resume` |
| `scripts/pseudo_label_ch2.py` | псевдоразметка «ч2/оталькованные» текущей моделью + contact-sheets для визуальной проверки; `--reject stem…` — отбраковка |
| `scripts/train_grain_cnn.py --prepare/--train` | CNN срастаний на weak labels |
| `scripts/calibrate_grains.py` | калибровка морфологических порогов |
| `scripts/sweep_rule.py` | свип порогов правила (зоны/фаза) по кэшу предсказаний |
| `scripts/evaluate.py` | все метрики: MAE талька (hold-out), негативные контроли, macro-F1 срастаний, accuracy сортов end-to-end |

Порядок воспроизведения модели талька v3 (важно: без ч2-позитивов ч2-негативы
учат сеть шорткату «тёмный тракт → не тальк»):

```bash
python scripts/make_talc_masks.py --k-dark 0.6                 # 39 уточнённых масок ч1
python scripts/train_talc.py --epochs 40 --negatives 60 --neg-domains ch1
python scripts/pseudo_label_ch2.py                             # + визуальная проверка листов
python scripts/pseudo_label_ch2.py --reject <плохие stem'ы>
python scripts/train_talc.py --epochs 20 --lr 1e-4 --resume models/talc_best.pt \
    --negatives 120 --neg-domains all --extra-dir data/talc_masks_ch2
python scripts/evaluate.py
```

## Метрики (hold-out, см. `outputs/evaluation.json`)

Семантика GT изменена на «фазу» (04.07), поэтому цифры не сопоставимы с v1 напрямую:

- тальк: MAE **4.75 п.п.** (фаза, hold-out ч1; v1 на зонах: 7.31);
- ложная «оталькованность» негативов: **0 %** фото с фазой > 10 % (v1: 6.7 %);
- домен ч2 (главный фикс): верных «оталькованных» **14/20** (v1: 5/20);
- классификация сорта end-to-end: accuracy **0.767**, macro-F1 **0.766**
  (потолок объясняется меткой на уровне ОБРАЗЦА при анализе одного кадра —
  часть кадров «оталькованных» проб реально бедна тальком);
- срастания: image-level macro-F1 **0.91** (CNN);
- производительность: панорама 300 Мп — **124 с** на RTX-GPU (норматив ТЗ ≤ 5 мин).

## Ограничения и известные допущения

- Метки классов — на уровне пробы, не кадра: единичный кадр может не содержать
  диагностических признаков своего класса (потолок e2e-метрики).
- Семейство снимков `1822101*` (третий тип съёмки, низкое разрешение) моделью талька
  не поддерживается — отбраковано из псевдоразметки.
- Плашка шкалы «300 мкм» на ч2 пока не маскируется (TODO — этап 0.3 плана).
- Изображения свыше ~1 Гп потребуют потокового чтения (pyvips).
- GeoJSON в пиксельных координатах (ось Y вниз); Shapefile — через `ogr2ogr`/QGIS.
- В git через LFS лежат только рабочие веса (`talc_best.pt`, `grain_cnn.pt`) — уложились
  в бесплатный лимит LFS (1 ГБ). Промежуточные чекпоинты итерации (`talc_v1_zonefill.pt`,
  `talc_v2_ch1refined.pt`, `talc_v2b_noch2neg.pt`, `talc_best_v1.pt`, `talc_last.pt`,
  `talc_v3_final.pt` — идентичен `talc_best.pt`) в репозиторий не включены; они
  воспроизводимы по шагам из раздела «Обучение и калибровка» выше.
